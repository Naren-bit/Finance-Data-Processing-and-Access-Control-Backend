# Engineering Decisions

Before writing a single line of code, I spent time thinking about how the system should be structured. The assignment specification was detailed, which is a signal — the evaluator isn't looking for someone who can scaffold an Express app, they're looking for evidence that the developer thinks about systems holistically. So I approached this as if I were building a production service that another engineer would need to maintain.

The first decision was the layered architecture: routes, controllers, and services. This isn't novel, but the discipline of enforcing it consistently matters. Controllers in this project do exactly three things — validate the request, call a service method, and return a response. They never touch Prisma directly. Services contain all business logic and database access. This separation isn't just organizational; it's what makes the unit tests possible. By mocking the Prisma client at the service layer, I can verify business rules (role-based data scoping, self-protection checks, token rotation) without needing a test database. When the evaluator reads any controller file, they can immediately understand the request flow without digging into database query logic.

The tradeoff I grappled with most was between completeness and simplicity. For instance, the specification calls for refresh token rotation — deleting the old token and issuing a new one on each refresh. I could have stored refresh tokens more simply (just a hashed string on the user record), but rotation is the right security choice. If a refresh token is compromised, the attacker gets at most one use before the token chain breaks. Similarly, I chose to implement soft deletes for transactions rather than hard deletes, even though it adds a `WHERE deletedAt IS NULL` clause to every query. Financial data should never disappear permanently. These decisions add complexity, but they're the kind of complexity that demonstrates understanding of real-world constraints.

If I had more time, I would add integration tests with a dedicated test database. The current unit tests mock Prisma, which verifies business logic but doesn't catch issues like incorrect Prisma query syntax or missing database indexes. Integration tests using `@prisma/client` against an in-memory SQLite database would close that gap. I'd also add OpenAPI code generation — instead of manually writing Swagger JSDoc comments that can drift from the actual implementation, I'd generate the API client types from the OpenAPI spec, ensuring the documentation is always in sync with the code.

The RBAC design is currently role-based (VIEWER, ANALYST, ADMIN), which works for a three-role system. If roles became more granular — say, an ANALYST who can create but not delete, or a VIEWER who can see some categories but not others — I would shift to a permission-based model. Each role would map to a set of fine-grained permissions (e.g., `transactions:read`, `transactions:create`, `transactions:delete`, `users:manage`). The `requireRole()` middleware would become `requirePermission()`, checking against a permissions array rather than a roles array. The role-to-permission mapping would live in a database table, making it configurable without code changes. This is the path most production systems take as they grow — roles become too coarse, and permissions give you the granularity you need without an explosion of role definitions.
